# Menu Item Enrichment — Schema ADR
**Date:** 2026-07-19  
**Repo:** `kwickly-api`  
**Domain:** Database & Schema  
**Status:** Implemented

---

## Context

The original `menu_items` table was minimal — it stored only the core commercial fields (`name`, `price`, `imageUrl`) and basic dietary flags (`isVeg`, `isJain`, `isGlutenFree`, `spiceLevel`). As we onboard real tenants (starting with Punjabi Chaska), we identified three gaps that are industry standard on all major food-ordering platforms:

1. **Item tagging / badging** — Zomato, Swiggy, and McDonald's all surface contextual signals (Bestseller, Chef's Special, New, etc.) that reduce decision fatigue and drive add-to-cart conversion.
2. **Nutrition & ingredient transparency** — FSSAI guidelines recommend calorie disclosure. Health-conscious users abandon menus that lack ingredient or allergen info.
3. **Enhanced availability** — the existing `availability` enum (`always` / `time_window` / `days`) was incomplete: there was no way to specify *which* days, or set an expiry date for limited-edition items.

---

## Decision

### Tag / Badge Fields (manual admin flags)

All tags are boolean flags managed by the restaurant admin. A future sprint will add computed auto-tagging from order analytics.

| Column | Type | Default | Purpose |
|--------|------|---------|---------|
| `is_bestseller` | `boolean` | `false` | ★ Most-ordered item |
| `is_chef_special` | `boolean` | `false` | 👨‍🍳 Chef's curated pick |
| `is_new` | `boolean` | `false` | 🆕 Recently added |
| `is_popular` | `boolean` | `false` | 🔥 High order velocity |
| `is_restaurant_special` | `boolean` | `false` | 🏠 Signature / house dish |
| `is_limited_edition` | `boolean` | `false` | ⏰ Time-limited offering |
| `is_healthy_choice` | `boolean` | `false` | 💚 Low-calorie / light option |

**Tag priority (UI):** Max 2 tags displayed per item to avoid badge clutter.  
Priority: `Bestseller > Chef's Special > Restaurant Special > New > Popular > Limited > Healthy`

### Nutrition Fields

| Column | Type | Purpose |
|--------|------|---------|
| `calories` | `integer` | kcal per serving (FSSAI-aligned) |
| `serving_size` | `text` | e.g. `"1 plate (350g)"` |
| `ingredients` | `text[]` | Array of ingredient names |
| `allergens` | `text[]` | Subset of FSSAI/EU-14 allergen list |
| `protein` | `numeric(5,1)` | Grams per serving |
| `carbs` | `numeric(5,1)` | Grams per serving |
| `fat` | `numeric(5,1)` | Grams per serving |

All nutrition fields are **nullable**. If `calories` is null, the storefront hides the nutrition row entirely.

### Enhanced Availability Fields

| Column | Type | Purpose |
|--------|------|---------|
| `available_days` | `text[]` | `['monday','wednesday','friday']` — null means all days. Used when `availability = 'days'` |
| `available_until_date` | `date` | Hard expiry date for seasonal items. API hides items past this date |

### Availability Resolution Logic (API layer)

```
Priority order (highest wins):
  1. status = 'HIDDEN'                         → never included in API response
  2. status = 'OUT_OF_STOCK'                   → returned with isAvailable: false
  3. availableUntilDate < today                → isAvailable: false, reason: 'seasonal_ended'
  4. availability = 'days' AND
       today not in availableDays              → isAvailable: false, reason: 'not_available_today'
  5. availability = 'time_window' AND
       now not between availableFrom-Until    → isAvailable: false, reason: 'outside_time_window'
  6. availability = 'always'                   → isAvailable: true
```

The API always returns the item in the response (never silently drops it), along with `isAvailable`, `availabilityReason`, and `availableFrom`/`availableUntil` so the storefront can show helpful "Available 12pm–3pm" hints.

## Consequences

- **Admin web** needs a "Tags & Nutrition" section in the menu item editor.
- **Storefront** `MenuClientView` needs badge rendering, nutrition display, and availability-aware ADD button state.
- **Seed data** for Punjabi Chaska is updated with realistic calories, ingredients, and tags for all items.
- Future: `is_bestseller` and `is_popular` will be auto-computed nightly from `order_items` aggregates and synced back as a scheduled job.
